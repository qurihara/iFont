# 2文字課題の音源となる合成エンジンの選定 (2026-06-26)

これは `next_phase_2char_audio.md` の「再開時の最初の作業 2」にあたる調査である。

## この作業で決めたいこと

競技かるた（百人一首）の読み上げに似た音声で、1〜2文字の短いかなを刺激として作りたい。とくに、競技かるたの読みに特徴的な「1文字目から2文字目にかけて音程が上がり、その後はほぼ棒読みになる」という韻律を、実験の独立変数として、付けたり付けなかったりできるように制御したい。そのうえで、次の条件を満たすエンジンを選ぶ。

- かなを全種類（清音・濁音・半濁音・拗音・促音・ゔ など）合成できること。
- 1文字目と2文字目それぞれの音程（ピッチ）を、合成のときに数値で指定できること。
- macOS（Apple Silicon）でローカルに動くこと。
- 多数の刺激（数百から千の規模）を、画面操作なしのスクリプトで一括生成できること。
- 合成した音声を、オンラインのクラウドソーシング実験で参加者に再生・配信してよいライセンスであること。

## 重要な前提の整理

この実験で本当に必要なのは「歌う音色」そのものではなく、モーラ（おおむねかな1文字）ごとの精密なピッチ制御である。競技かるたの「1→2文字目の音程上昇」は、各モーラのピッチを個別に設定できれば再現できる。この観点に立つと、候補は大きく2系統に分かれる。

- 系統A（ピッチを制御できる話声合成）。VOICEVOX とその同系（COEIROINK・SHAREVOX）は、合成の前にモーラごとのピッチを数値で上書きでき、画面操作なしの REST API で一括生成できる。無償で Apple Silicon でも動く。本実験の「ピッチ上昇のあり・なし」という独立変数の操作に最も素直に合う。
- 系統B（歌声合成）。NEUTRINO・OpenUtau・Sinsy・NNSVS などは、楽譜の音符に音高を割り当てて歌わせる。競技かるた特有の「節回し（歌うような持続）」の音色まで忠実に再現したい場合に向く。ただし一括自動生成のしやすさや Apple Silicon 対応はエンジンによって差がある。

加えて、作業1で扱った Kikiwake の実際の読み上げ音声は、競技かるたの読み手そのものの声であり、生態学的な妥当性が最も高い。これは合成ではない第3の音源として、比較条件に使える（ただし作業1の結果のとおり、一般日本語の連接のうち押さえられるのは出現頻度で約半分である）。

評価軸は、ピッチ制御の細かさ、かなの網羅、Apple Silicon でのローカル動作、画面操作なしの一括生成、オンライン公開実験での配信の可否（ライセンス）である。

## 推奨

本実験には、次の組み合わせを推奨する。

1. 主軸は VOICEVOX を中心とする系統A（ピッチを制御できる話声合成）とする。理由は次のとおりである。第一に、モーラごとのピッチを合成前に数値で上書きできるので、「1→2文字目の音程上昇」を独立変数として、上昇あり・上昇なしを正確に作り分けられる。第二に、REST API により画面操作なしで数百から千の刺激を一括生成でき、刺激プールの再現性が高い。第三に、Apple Silicon でローカルに動き、無償である。第四に、本プロジェクトには既に聴覚実験のページ（experiment/audio.html）と刺激生成の仕組み（make_audio_stimuli.py 系）があり、現在の macOS の say による音源を VOICEVOX の REST API に置き換えるのは自然な拡張である。
   - 注意点を1つ述べる。VOICEVOX の pitch の数値は絶対的な周波数（ヘルツ）ではない。実験では、設定した pitch 値と、実際に合成された音声の基本周波数（Praat や parselmouth で測る）との対応表を作って較正する必要がある。これにより、上昇量を半音などの物理量で管理できる。
   - 候補のうち COEIROINK は Apple Silicon にネイティブ対応し、API でモーラごとのピッチを設定できるため、VOICEVOX の有力な代替である。声の質感や話者の選択肢を比べて決めるとよい。

2. 競技かるた特有の「節回し」の音色まで効果として確かめたい場合の比較用として、系統B（歌声合成）から少数の刺激を作る。Apple Silicon にネイティブ対応し、ピッチ曲線を手で直接編集できる OpenUtau が扱いやすい。音質を最優先するなら NEUTRINO も候補になる。これは「韻律の手掛かりが、抽象的な音程上昇だけで説明できるのか、それとも歌うような音色まで含めて効くのか」という研究上の問いに対応する比較である。

3. Kikiwake の実際の読み上げ音声は、生態学的な妥当性が最も高い第3の音源として、引き続き比較条件に使う。これは作業1で射程を定量化済みである。

商用エンジンは、現時点では見送る。CeVIO AI は Windows 専用で歌唱の自動化ができない。A.I.VOICE2 は「素材として配布」を禁じており、オンラインでの配信に懸念がある。Synthesizer V はピッチ制御の精度が最も高いが、有償であり、画面操作での書き出しになる。系統A で要件を満たせる見込みが高いため、まず無償で自動化しやすい系統A から着手する。

## ライセンス上の運用方針（全エンジンに共通）

調査の結果、どのエンジンも「オンライン公開実験で合成音声を配信してよいか」を利用規約で明示してはいなかった。そこで、次の運用を全エンジン共通の安全策とする。

- 刺激音声は、実験の中で再生する用途に限る。参加者が刺激クリップを再利用可能な素材としてまとめてダウンロードできる形では公開しない。
- 使用するエンジンと話者の規定のクレジットを表示する（VOICEVOX なら「VOICEVOX：話者名」の形式）。使用する話者ごとの個別規約も確認する。
- 論文発表や刺激の公開配布を予定する場合は、採用するエンジンと音源の権利者に、事前に書面で確認する。
- なお、本プロジェクトは既に macOS の say による合成音声を聴覚実験で配信しており、同じ配慮が必要である点も記録しておく。

## 次の作業への接続

作業3として、競技かるたに限定した2文字課題の試作に進む。試作では、VOICEVOX をローカルに起動し、清音46字の範囲で、先行する1文字目から、時間でゲートしたターゲットの2文字目へ進む刺激を作る。1→2文字目の音程上昇を、あり・なしの条件として用意する。回答は既存の50音グリッドを使う。較正のために、設定したピッチ値と実際の基本周波数の対応表も作る。
