# 競技かるた限定 2文字課題の試作 (2026-06-26)

これは `next_phase_2char_audio.md` の「再開時の最初の作業 3」にあたる試作である。作業2で主軸に決めた VOICEVOX を実際にローカルで動かし、2文字課題の刺激を作る中核の技術的な確かさを検証した。

## やったこと

VOICEVOX の音声合成エンジン（macOS Apple Silicon ネイティブ版、バージョン 0.25.2）をローカルに用意し、画面操作なしの REST API で刺激を生成する仕組みを作った。統一モデルの1試行は「先行する文脈 C1 から、ターゲット C2 へ進み、C2 を時間でゲートする」である。ここでは C1 を1文字目、C2 を2文字目として、清音の範囲で作った。

生成の手順は次のとおりである。各かなを単体で `/audio_query` に投げ、正しいモーラ（子音と母音の音素、およびその長さ）を取り出す。それを2つ並べて2モーラの発話を組み立てるので、辞書にない任意のかな対でも誤読しない。各モーラのピッチを上書きして、競技かるた風の「1→2文字目の音程上昇」を、あり（rise）と なし（flat）の2条件で作り分ける。合成したあと、組み立てたクエリのモーラ長から2文字目の開始時刻を厳密に計算し、2文字目だけを `frac%`（0から100）で時間ゲートする。1文字目は常に全部提示する。合成なので境界が既知であり、Montreal Forced Aligner のような外部の整列は不要である。

## 検証して分かったこと

### 時間ゲートは厳密に機能する

2文字目の開始時刻と長さは、合成クエリのモーラ長から正確に求められる。実際に、「きの」で frac を 0・50・100 と変えると、音声の全長がそれぞれ「1文字目のみ」「1文字目＋2文字目の前半半分」「1文字目＋2文字目の全部」に対応して変化した。Montreal Forced Aligner を使わずに、2文字目を任意の割合で切り出せることを確認した。

### ピッチは制御でき、flat と rise を明確に作り分けられる

合成した音声の基本周波数を Praat（parselmouth）で測ると、flat 条件はほぼ平坦（上昇およそ0半音）、rise 条件は2文字目が明確に上がった。たとえば「あき」では、flat は1文字目も2文字目も約106ヘルツで平坦、rise は2文字目が約150ヘルツに上がった（約6半音の上昇）。「わた」でも同様に、rise は2文字目が約147ヘルツに上がった。

### 較正でわかった、設定値と実際の上昇量の関係

較正の結果、VOICEVOX のモーラの `pitch` は基本周波数の自然対数 ln(F0) であった。つまり F0[ヘルツ] = exp(pitch) で、半音（周波数比 2 の 12 乗根）の上げ幅は Δpitch = ln(2)/12 ≈ 0.0578 にあたる。

ただし、合成のニューラルボコーダが、短い2モーラ区間の小さなピッチの振れを圧縮することが分かった。8つのかな対で設定値を振って実測したところ、次のようになった。

| 設定の上げ幅 | 実測の上げ幅（8対の平均±標準偏差） |
|---|---|
| 0 半音 | -0.1 ± 0.9 半音 |
| 2 半音 | +0.2 ± 1.2 半音 |
| 4 半音 | +1.2 ± 1.2 半音 |
| 6 半音 | +3.0 ± 1.6 半音 |
| 8 半音 | +6.0 ± 1.1 半音 |
| 10 半音 | +8.1 ± 1.0 半音 |
| 12 半音 | +10.0 ± 0.7 半音 |

設定が0から4半音の範囲は大きく潰れて、ほとんど上昇しない。設定が8半音を超えるあたりからは、ほぼそのまま実測に反映される。設定8半音で実測およそ6半音（5度弱）の、競技かるたらしい上昇が頑健に得られる。そこで生成スクリプトの既定値は設定8半音にした。

注意点として、かな対による分散が標準偏差で1から1.5半音と大きい。すべてのかな対で実測の上昇量を厳密に一様にしたい場合は、対ごとに設定値を合わせ込む較正をするか、合成後に WORLD ボコーダなどでピッチを目標の輪郭に置き換える再合成をするのが確実である。明確な「上昇あり・なし」の対比を作るだけなら、設定8半音で十分に区別がつく。

## 話者について

試作では、競技かるたの読み手に近い、明瞭な男性の声として「玄野武宏 / ノーマル」（スタイル番号11）を使った。VOICEVOX の声はキャラクターの声なので、競技かるたの読み手そのものの声質とは異なる。読み手の声質まで重視する場合は、生態学的な妥当性が最も高い Kikiwake の実際の読み上げ（作業1で射程を定量化済み）を比較条件として併用する。

## 出題するかな対の確定と刺激プールの生成 (2026-06-26 追記)

### 対象のかな対

出題するかな対は、百人一首の上句に実在する、競技かるたの清音46字どうしの隣接対に絞った。上句100首の現代かなを連続読み（空白を除く）として隣り合うかな2文字を取り出し、両方の文字が清音46字に入るものだけを集めた。結果は次のとおりである。

- 異なり対は632種、延べ1270回。最も多い対は「やま」の14回で、「しの」13回、「きの」「もの」11回と続く。
- 632種のうち338種は1回しか現れず、294種は2回以上現れる。
- ターゲットになる2文字目は46字中45字が現れ、先行する1文字目も45種が現れる。

一覧と頻度は `two_char_audio/hi_karuta_pairs.json` に保存した。

### プールの作り方（時間ゲートは再生時にブラウザ側で行う）

全部の frac をあらかじめ焼き込むと、632対 × 2条件 × 21段階で約26,500ファイルになり過大である。そこで、対と条件（flat・rise）ごとに全長の音声を1つだけ作り、2文字目の時間ゲートは再生時にブラウザ側で行う方式にした。これは既存の `pilot_audio` のライブ切り出しと同じ仕組みで、プールは 632対 × 2条件 = 1,264ファイルで済み、frac を実行時に柔軟に変えられる。

ファイル名は秘密の値（SECRET_SALT）でハッシュ化し、公開する manifest には正解（2文字目）を含めない。2文字目の開始時刻と長さだけを記録し、ブラウザ側のゲートに使う。正解は別の answer_key に分けて保持し、Google Apps Script 側に置く。

生成結果は、1,264ファイル・合計約9メガバイト（mp3）であった。

トレードオフを述べる。全長の音声がファイルに入るので、低い frac の試行でも、音声を取り出せば2文字目の全部を聴けてしまう。チート耐性はあらかじめ焼き込む方式より下がる。本プロジェクトの不正対策は標準セットの方針なので当面はこれで進め、本番公開の際にあらかじめ焼き込む方式へ切り替えることもできる。

### プールの品質確認

合成した音声の基本周波数を Praat で測り、設計どおりの輪郭になっているかを確かめた。rise 条件を約84対、flat 条件を約42対、サンプルして測った。

- rise 条件の実測の上昇量は、平均 +4.8半音、中央 +5.0半音。3半音以上の明確な上昇が得られた対が83%で、上昇が2半音に満たない対は7%であった。
- flat 条件はほぼ平坦で、上昇量の絶対値の中央は0.6半音であった。

大半の対では狙いどおりの「上昇あり・なし」の対比が得られている。一部（おおむね1割弱）は、1文字目が「き」のように高めの母音を持つ場合などにボコーダの圧縮が強く出て、上昇が弱い。これらは、本番のプールでは対ごとに設定値を合わせ込む較正をするか、WORLD ボコーダでピッチを置き換える再合成をして揃えるのがよい。

成果物は `two_char_audio/build_2char_pool.py`（プール生成）、`two_char_audio/hi_karuta_pairs.json`（対の一覧）。生成した音声と answer_key は容量と秘匿のため Git の管理対象外とした。

## 2文字課題の実験ページ (2026-06-26 追記)

2文字課題の実験ページを作り、ブラウザで動作を確認した。既存の聴覚実験ページ（`experiment/audio.html`）と並ぶ形で、`experiment/audio2char.html` と `experiment/audio2char.js` を新しく作った。

仕組みは次のとおりである。各問で、ひらがなを2文字つづけて読み上げる。1文字目は最後まで聞こえ、2文字目は途中までしか流れないことがある。回答は既存の固定50音グリッド（競技かるた46字）で、2文字目だと思う1文字を選ぶ。容易な統制試行は2文字目を最後まで流す（frac=100）。

2文字目の時間ゲートは、再生のときにブラウザ側で行う。`pilot_audio` と同じ Web Audio の仕組みで、合成音声をデコードし、ファイルの先頭から「2文字目の開始時刻 + frac% × 2文字目の長さ」までを、末尾に短いフェードを付けて再生する。2文字目の開始時刻と長さは、公開する manifest に入れてある（正解の文字は入れていない）。正解は別の answer_key に分けて持ち、Google Apps Script 側で採点する。

刺激プールは `experiment/audio2char_stimuli/` に置いた（632対 × flat・rise = 1,264ファイル）。公開する manifest は `experiment/audio2char_manifest.json`、非公開の answer_key は `experiment/answer_key_2char.json`（Git の管理対象外）である。

ローカルのブラウザで、同意・説明・練習まで通して確認した。50音グリッド（競技かるた46字）が正しく並び、合成音声がデコードされて2文字目がゲート再生され、回答すると条件（flat か rise）・上げ幅・frac・選んだ文字・反応時間などが正しく記録された。コンソールにエラーは出なかった。
