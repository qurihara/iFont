# 本番のセッション設計と予算配分(F1・確定案)

作成: 2026-07-18。F2(干渉判定の事前登録ルール、docs/prereg_interference.md)の結論を所与として、
Yahoo!クラウドソーシングでの本番実施のセッション構成・人数・謝礼・予算配分を確定する。
総予算は10万円以内で、予備費を約3割残す主案と、予備費約1割の代替案Bを併記する。

## 1. Yahoo!クラウドソーシングの実勢(調査結果と出典)

### 1.1 依頼側の料金体系(公式)

オーナー(依頼者)向け公式ページに次の料金が明記されている。

| 項目 | 料金 | 備考 |
|---|---|---|
| 基本料金 | 10円/1タスク | 1タスク=設問セット(最大99問)を1人が完了するごと |
| 謝礼ポイント追加 | 1.5円/1ポイント | 参加者に渡す1ptにつき依頼者は1.5円を支払う |
| 属性指定(性別・年齢・地域) | 無料 | |
| ポイント付与そのもの | 無料 | 付与手続きに追加費はない |
| 最低利用料 | 1,000円/1掲載(税別) | 掲載単位の下限 |
| 消費税 | 上記すべてに別途10% | |

つまり **参加1人あたりの依頼側コスト = (10円 + 1.5円 × 謝礼pt) × 1.10** である。
謝礼150ptなら1人258.5円、130ptなら1人225.5円になる。参加者側の手数料はなく、
1pt=1円相当のPayPayポイントがそのまま渡る。

出典:
- 料金(公式オーナー向け): https://crowdsourcing.yahoo.co.jp/special/owner/what
- 利用価格の案内(公式ヘルプ): https://support.yahoo-net.jp/SccCrowdsourcing/s/article/H000009403
- 報酬の仕組み(公式ヘルプ・参加者側): https://support.yahoo-net.jp/PccCrowdsourcing/s/article/H000005778

### 1.2 謝礼の相場

- プラットフォーム全体の謝礼は1タスク5〜300ptに分布し、簡単なアンケートは10〜30pt、
  専門的な調査は100pt以上が多い(利用者レポート)。
- 実例として「Web上の心理学実験・約10分・PC限定・離席不可」で **134pt** の案件が確認できる。
  本計画の課題(10〜14分・PC推奨・外部ページで完了コード方式)はこの型とほぼ同じである。
- 研究利用の解説記事では「クラウドソーシングのアンケートは10分あたり100円またはそれ以下が多い」
  とされ、報酬設定は倫理審査と相談のうえ決めるべきとされている。

出典:
- 研究利用の解説(アカリク): https://acaric.jp/articles/5029
- 利用者による謝礼帯の報告: https://zaitakushigoto.com/298286772/
- 特徴解説(手数料なし・謝礼帯): https://mobinc.jp/column/2026/04/01/understanding-yahoo-cloudsourcing-features-reputation-and-paypay-points/
- 案件例(タスク一覧・詳細): https://crowdsourcing.yahoo.co.jp/request/detail/3589566058 ほか

### 1.3 最低賃金への配慮

2025年10月改定の最低賃金は全国加重平均1,121円、東京都1,226円である。
本計画の謝礼は約720〜780円/時に相当し、プラットフォームの相場(10分100円=600円/時が多い、
心理学実験の実例134pt/10分=約800円/時)の上寄りに合わせたが、雇用の最低賃金には届かない。
クラウドソーシングの謝礼は雇用契約の賃金ではないため法的な問題はないが、
倫理審査にはこの比較を明示して判断を仰ぐ(未決事項(4))。予算10万円で全課題を最低賃金相当
(約2倍の単価)にすると必要額が約12万円を超え、両立しない。

出典:
- 東京都公報(1,226円): https://www.koho.metro.tokyo.lg.jp/2025/10/oshirase03.html
- 全国一覧(加重平均1,121円): https://www.hrm-co.jp/knowledge/national-minimum-wage/

## 2. 現行実装の確認(設計の前提)

セッション時間の見積もりは、実装のパラメータと再生系列から次のとおり組み立てた。

### 2.1 乙課題(Stage 1、pilot_soa_audio.js / pilot_soa_visual2.js)

- 現行: SOA7水準 {50,83,133,200,300,450,700}ms × PER_LEVEL=6 = 42問 + 練習2問。
  出題は混ぜたリストから順に配る方式で、同一セッション内の同字繰り返しはない。
- 1問は「3音(3字)の連続提示 + 1番目と2番目の回答(50音グリッドから2回選択)」で約12〜15秒。
  実装の教示にも「所要8〜12分」とあり、PIのパイロット実測もこの範囲に収まる。
  同意・練習・音量確認を足して **1セッション約10〜12分** と見積もる。
- F2の結論により42問構成は変更しない(水準あたり9問化は費用がほぼ同額のまま15分を超えるため)。

### 2.2 frac課題(Stage 2、audio1char.js / visual1char.js / audio2char.js)

- **FRAC_GRID は3課題とも {0,5,10,…,100} の21水準**(現行実装で確認)。frac=100の5%を
  キャッチ(統制)試行とし、fracはグリッドから一様に割り当てる実装である。
- 聴覚1文字: 音の系列が「開始合図音0.08s + 間0.3s + 刺激最大0.2s + 間0.5s + 終了合図音2回(0.22s)」
  で固定部が約1.1〜1.3秒。72字グリッドからの選択に2.5〜4秒。**1試行平均5秒**(4〜6秒)と置く。
- 聴覚2文字: 刺激が「C1全提示 + C2打ち切り」で c2_onset=0.30s(マニフェスト全対で一定)、
  再生は最大0.5秒。合図音込みで固定部約1.4〜1.6秒。**1試行平均5.5秒**(4.5〜6.5秒)と置く。
- 視覚1文字: 提示は最大0.2秒、78字グリッドからの選択に3〜4秒。**1試行平均4秒**(3〜5秒)と置く。
- 聴覚の刺激プールはマニフェスト上72音ある。確定済みの設計(を・ぢ・づ・ゔは同音のため除外)に
  合わせ、本番は **68音に絞る**(回答グリッドは72ボタンのままにし、同音クラスは正解扱いにする)。
  視覚1文字は78字、乙の視覚は72字である。
- 視覚1文字の提示アルゴリズムは現行 ALGO_LIST=["fade","blur","moya"] の3種ランダムだが、
  本番は **1種に固定する**(3種のままだと字ごとの観測が3分割され、必要人数が3倍になる。
  変換gの応用でも表示方式は1つに決める必要がある)。どの1種にするかはPIのfracパイロット
  体験をふまえて確定する(未決事項(2)。仮置きは fade)。

## 3. Stage 2 の観測数の目標(根拠)

**目標: 字(音)ごとに約60〜90観測を21水準に散らす(セルあたり平均3〜4観測)。**

根拠は次の3点である。

1. **F2の較正との整合**: F2の検出力計算は「Stage 2の字ごと推定の標準誤差は n=40〜60 で
   6.5〜8ポイント」を基準に δ を決めた。字あたり60観測はこの基準の上端であり、
   主案の88観測なら標準誤差は約5.4ポイントまで下がる(SEが1/√nに比例するとして換算)。
2. **部分プーリングの節約**: 推定するのは字ごとの心理測定関数(頭打ちパラメータ付きの
   2〜4自由度の滑らかな曲線)であり、セルごとの独立推定ではない。階層モデル
   (字・参加者・水準の部分プーリング)では、セルあたり10〜20観測を要する独立推定と違い、
   セルあたり3〜5観測(=字あたり60〜100)で曲線が安定して引けることが知られている
   (GLMMによる心理測定関数推定: Moscatelli, Mezzetti & Lacquaniti, 2012, J Vis;
   ベイズ的psignifitの少数試行推奨: Schütt, Harmeling, Macke & Wichmann, 2016, Vision Res)。
3. **予算の上限**: セルあたり10観測(字あたり210)を狙うと聴覚1文字だけで延べ約1.4万観測
   =100人超になり、1課題で3万円を超える。総額10万円では3課題が成立しない。

出題は現行の「試行ごと一様乱択」から「デッキ配り」に変える(乙で確立済みの方式の移植)。
字は非復元で均等に(1人あたり各字ほぼ2回)、fracは21水準のタイルを繰り返しシャッフルして
割り当てると、セル観測数の偏り(乱択ではポアソン的に散らばり、空セルも出る)を抑えられる。

## 4. セッション設計(主案)

方針: Stage 1は較正のゲートであり、論文の主結果はStage 2の字ごとの曲線と変換gである。
そこでStage 1はF2の予算代替案(付録A: δ=8pt、聴覚60・視覚40)を主ルールとして小さく張り、
浮いた予算をStage 2の精度(字あたり88観測)と予備費(約3万円)に回す。予備費は
「Stage 1が保留になったときの増員」または「視覚で干渉ありのときの視覚2文字課題」に使う。

各課題の構成は次のとおり。有効数は除外(15%を見込む)後の人数、募集枠は ceil(有効数/0.85)。

| 課題 | 1人あたり試行 | 所要 | 謝礼 | 有効数 | 募集枠 | 観測の目標 |
|---|---|---|---|---|---|---|
| S1 聴覚乙 | 練習2 + 本番42(7水準×6) | 約10〜12分 | 130pt | 60 | 71 | 干渉判定(δ=8pt): 真の差0を「干渉なし」と正しく判定86.5%、8pt劣化の検出84.4% |
| S1 視覚乙 | 練習2 + 本番42(7水準×6) | 約10分 | 130pt | 40 | 48 | 同上: 94.4%、88.2% |
| S2 聴覚1文字 | 練習5 + 本番140(うちキャッチ7) | 約14分 | 170pt | 45 | 53 | 68音×(133×45)で字あたり88観測、セルあたり4.2、字ごとSE約5.4pt |
| S2 視覚1文字 | 練習5 + 本番160(うちキャッチ8) | 約12〜13分 | 150pt | 45 | 53 | 78字で字あたり88観測、セルあたり4.2(アルゴリズム1種固定が前提) |
| S2 聴覚2文字 | 練習5 + 本番120(うちキャッチ6) | 約13〜14分 | 170pt | 40 | 48 | C2の68音で字あたり67観測、セルあたり3.2、字ごとSE約6.2pt |
| (条件付き) 視覚2文字 | 視覚乙で「干渉あり」のときのみ | 約13分 | 150pt | 40 | 48 | 予備費から支出 |

- 判定確率はF2のモンテカルロ結果(power_sim_interference_results.json)の該当セルの値である
  (聴覚N=60・δ=8: p_no=0.865/p_yes=0.844。視覚N=40・δ=8: p_no=0.944/p_yes=0.882)。
- 謝礼の時給換算は720〜780円/時で5課題ほぼ一定にした(相場の上寄り。1.3節参照)。
- セッション時間の内訳例(聴覚1文字): 本番140×5秒=11.7分 + 練習・同意・音量確認で約2分。

### 4.1 Stage 1 を小さくすることの代償と回収

δ=8ptの主ルールは、F2の主案(δ=5pt)より粗いゲートである。一様な5〜8ptの劣化は
「保留」に落ちやすくなる。この代償は次の2段構えで回収する。

- **保留になった場合**: 予備費でF2主案の水準(有効130/80、募集150/95)まで増員し、
  全データでδ=5ptの判定をやり直す。増員の費用は聴覚+79人・視覚+47人で28,413円であり、
  予備費30,024円に収まる(増員後の総支出は98,390円で総予算内)。
  二段階判定になるため、αの配分(たとえば各段α=0.025)をF2のスクリプト再実行で
  確定してから事前登録する(未決事項(1))。
- **Stage 2の精度向上との整合**: 主案では字あたり88観測(SE約5.4pt)まで精度が上がるため、
  「SEより小さい一様劣化はgを変えない」というF2の論法での閾値は5pt寄りが整合的である。
  δ=8はあくまで低予算ゲートの閾値であり、判定が割れたときに増員で5ptに戻せる設計とした。

### 4.1b 頭打ちが低い参加者への頑健性(2026-07-22 追記)

共同研究者パイロット(2人目の観測者)で聴覚の頭打ち58.3%を観測した。これを受けて
頭打ち60〜65%でのδ=8ゲートの判定確率を追加計算した(experiment/tools/
power_sim_lowplateau_cells.py、詳細はF2文書の付録A補足)。結論: 頭打ちが70%→60%に
下がっても「干渉なし」判定力は85.4%→81.9%(−3.5pt)、8pt劣化の検出は83.6%→81.5%
(−2pt)にとどまり、誤判定率は名目5%に制御されたまま。主案の構造(小さく張って
保留なら増員)は低頭打ちでも成立する。なお増員先をδ=5でなくN=80のδ=8に変える案は、
どの頭打ち水準でも判定力約90〜93%とやや高いが、Stage 2の精度(SE約5.4pt)との整合は
δ=5への増員のほうが良いため、既定の増員パス(δ=5・130/80)を維持する。
また同パイロットの視覚は頭打ち12/12=100%・S=200も6/6=100%で、視覚の想定90%が
保守的であることを実測で裏づけた(視覚N=40の主案を変える理由はない)。

### 4.2 品質管理(キャッチ試行と除外)

- frac3課題はキャッチ(frac=100)を固定数で混ぜる(7/8/6問)。実装は現行の確率5%でなく
  固定数にする(除外基準を決定的にするため)。
- 除外基準: F2の事前登録の機械判定(完了コード不一致、worker_id重複、機械的回答、所要秒、
  聴覚課題でのタッチ端末)に加え、fracでは **キャッチ正答率50%未満を除外** とする。
  まじめな参加者(全提示の正答率85%と仮定)が7問中3問以下になる確率は1.2%であり、
  誤除外はほぼ生じない。除外分は募集枠の15%上乗せで吸収する。
- 乙課題の除外はF2の規定どおり(キャッチは置かず、頭打ち水準の成績とRT・完了記録で判定)。

### 4.3 重複参加の扱い

- **同一課題内は1人1回**(worker_id重複と完了コードで機械除外。F2の規定どおり)。
- **課題間の重複参加は可**とする。解析は課題ごとに独立で、参加者ランダム効果は課題内でしか
  使わないため推定は偏らない。音声課題の経験者はヘッドホン環境が確認済みという利点もある。
  ただし同時掲載での疲労と取り違えを避けるため、掲載は課題ごとに時期をずらす(5節)。
- Yahoo!クラウドソーシングのホワイトリスト・ブラックリスト機能で「乙の完了者にfracを
  優先案内する」運用もできるが、必須にはしない(募集が遅い場合の加速手段とする)。

## 5. 予算表

単価の式: 1人あたり = (10 + 1.5×謝礼pt) × 1.10(消費税込み)。掲載ごとの最低利用料

### 削減の優先順位(募集が予算を超過しそうな場合)

1. 聴覚2文字の有効数を40から30へ(募集36、約3,500円の削減。字あたり50観測に低下)。

3. 乙の増員(予備費使途1)を聴覚のみに限定する(視覚は主案Nで確定させる。
   視覚はN=40でも判定確率94.4%と高いため影響が小さい)。

## 6. 実装に差し込む定数(推奨値)

```js
// ---- Stage 1 乙(pilot_soa_audio.js / pilot_soa_visual2.js、?prod=1) ----
const PER_LEVEL  = 6;    // 7水準×6=42問(現行どおり変更なし)
const N_PRACTICE = 2;    // 現行どおり

// ---- Stage 2 聴覚1文字(audio1char.js) ----
```

## 8. 検証の記録

- 料金体系は公式オーナー向けページの記載(10円/タスク、1.5円/pt、税別)から採り、
  1人単価・小計・総額をスクリプトで再計算して確認した(全行の合計69,976.5円、
  予備費30,023.5円、増員分岐後98,389.5円、いずれも10万円以内)。
- 判定確率はF2の power_sim_interference_results.json の該当セル(聴覚N=60/δ=8、
  視覚N=40/δ=8、代替案Bは聴覚N=130/視覚N=80/δ=5)をそのまま引用した。
- 字あたり観測数は (本番試行−キャッチ)×有効数÷字数 で計算した(聴覚1文字133×45/68=88.0、
  視覚1文字152×45/78=87.7、聴覚2文字114×40/68=67.1)。SEの換算はF2の較正
  (n=40〜60で6.5〜8pt)から1/√n則で外挿した(88観測で約5.4pt、67観測で約6.2pt)。
- セッション時間は実装の再生系列(合図音・間隔の定数、c2_onset=0.30s一定)と
  グリッド選択の想定(2.5〜4秒)から積み上げ、乙は教示記載(8〜12分)とPI実測で裏づけた。
- キャッチ除外の誤除外率(まじめな参加者p=0.85が7問中3問以下になる確率1.2%)を
  二項分布で計算して確認した。
