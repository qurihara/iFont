# 設計改訂 (2026-07-02)

議論を踏まえた実装の軌道修正を記録する。倫理審査書類は提出済みであり、今後この改訂によって編集はしない。

## 改訂の要点

1. **早押しクイズへの応用可能性を重視する。** これまで競技かるた優先としてきたが、語彙やことばのつながりを競技かるたの語彙に限定することはしない。
2. **視覚・聴覚とも2文字の課題にする。** 2文字目を当てる問題とする。1文字目はランダムな文字、2文字目もランダムな文字とする。
3. **通常ありえない文字の接続をどう扱うかは要検討**（後述の暫定方針を参照）。
4. **視覚は静止画ではなく時間的に変化する提示にする。** PowerPoint のアニメーションにあるような、時間的な移動・拡大縮小・透明度変化などをともない、時間とともに認識度が上がるアルゴリズムを複数検討する。フォントの種類も引き続き検討事項として留意する。
5. **視覚の長文提示の型式**は従来どおり「ある領域を固定して、その領域で文字が次々に表示される」とする。次の文字に移るとき、前の文字は一瞬で消去する。前の文字が目に残像として残ることが、2文字課題で重要な点になる。
6. **提示速度は競技かるたのルールに基づき、1文字 0.2 秒とする。**
7. **提示音高は競技かるたのルールに基づき、1文字目を B3（246.94ヘルツ）、2文字目を E4（329.63ヘルツ）に固定する。** これまでの「音程上昇のあり・なし」という独立変数はやめ、固定の設計とする。

## 方法論の判断（この改訂と同時に議論したこと）

### ボタン押し（反応時間）方式について

「分かったらボタンを押して回答し、単純反応時間を引き算する」方式（Kikiwake と同じ発想）は考えられるが、主軸は断片音声課題（2文字目を frac% まで聞かせるゲーティング）のままにする。理由は、フォント作成の核である変換関数 g には「明瞭度を横軸にした認識率の曲線」が必要で、断片音声課題はそれを直接与えるからである。ボタン押し方式は時間軸での識別時点の分布を与えるもので、連続ストリームでの検証アームとして後段に置く。クラウドソーシングでの反応時間測定は端末差でノイズが大きく、単純反応時間のベースラインブロックを設けても実験室より精度が落ちる点にも留意する。

### QUEST 法のような適応的な試行数最適化について

今回の目的には固定グリッドとロジスティック回帰の組み合わせが適切で、適応法は採らない。理由は次の3つである。

1. 推定したいのが閾値1点ではなく曲線全体だから。フォント応用では認識率の任意の点で逆引きするため、傾きや裾を含む曲線全体が要る。QUEST は閾値付近に試行を集中させる分、曲線全体の情報が薄くなる。
2. クラウドソーシングと相性が悪いから。適応法は試行ごとに正誤の即時判定が要るが、正解をクライアントに持たせると answer_key が漏れ、サーバに問い合わせると往復遅延と Google Apps Script の呼び出し上限が問題になる。参加者ごとに刺激系列が変わると併合解析も複雑になる。
3. participants を増やして cell あたりの観測数を稼ぐ方が、集計も再現も素直だから。

したがって、実装済みの2パラメータのロジスティック回帰（γ は選択肢数から固定、α と β を最尤推定）で足りる。改善は適応法ではなく「グリッドの置き方」で行う。パイロットの結果で情報量の多い frac 帯に段階を密に置く、静的な最適実験計画が副作用のない代替になる。なお、完成したフォントをユーザーごとに較正する応用の場面（初回設定で閾値を短時間で求める）では、1人・少試行・閾値1点という条件が揃うため、QUEST 的な適応法がまさに適切である。研究の測定と応用の個人較正で道具を使い分ける。

## 聴覚側の実装（この改訂で実施）

### 話者の選定（実測にもとづく）

絶対音高 B3・E4 を VOICEVOX で実現できるか、話者5種で実測した（`two_char_audio/verify_b3e4.py`）。男声2種（玄野武宏・青山龍星）は E4（330ヘルツ）に届かず破綻した。女声では四国めたんが最良で、1回の補正後の平均絶対誤差が14セント（半音の14%）だった。波音リツが20セントで次点。**話者は四国めたん（番号2）に決めた。** B3 から E4 への上昇（完全4度）は女声の自然な音域に収まる。

### 刺激プールの生成（`two_char_audio/build_2char_pool.py` を v2 に全面改訂）

- 対象は全72字（音声で区別可能なかな）× 全72字の総当たり 5,184 対。競技かるたの語彙に限定しない。
- 音高は1文字目 B3・2文字目 E4 に固定。VOICEVOX の mora.pitch は ln(F0) なので設定値は ln(246.94) と ln(329.63)。ファイルごとに合成後の基本周波数を実測し、50セントを超えるずれは1回補正して作り直す。実測値は answer_key に記録する。
- 提示速度は1文字 0.2 秒。子音長は自然値を残し、母音長で合計 0.2 秒に合わせる。動作確認で、2文字目の開始が正確に 0.3 秒（前余白 0.1 秒 + 1文字目 0.2 秒）、長さが 0.2 秒になることを確かめた。
- 2文字目の時間ゲートは従来どおり再生時にブラウザ側で行う。

### 通常ありえない連接の暫定方針

全対をプールに含め、Tatoeba コーパスでの連接頻度を共変量として answer_key に記録する。除外や層別は、あとから出題側（クライアントのフィルタ）や解析側で選べる。参考として、72字どうしの連接 5,184 種のうち、Tatoeba に1回も現れないものはおよそ1割強である。教示では「2文字のつながりに意味はありません。ことばとして自然かどうかは気にせず、聞こえた音だけで答えてください」と明示する。

### 実験ページの更新

`experiment/audio2char.js` を新仕様に合わせた。回答グリッドを競技かるた46字から全72字（濁音・半濁音・ゔを含む、audio.js と同じ配列）に変更し、条件の記録を pitch_scheme = "B3-E4" に一本化し、教示に「つながりに意味はない」ことを追記した。

## 視覚側の実装（これから）

方針だけ記録する。実装は次の段階で行う。

- 型式は「固定領域に文字が次々に表示される」。1文字 0.2 秒。次の文字に移るとき前の文字は一瞬で消去する（残像が2文字課題の重要な測定対象になる）。
- 1文字の 0.2 秒の中で、時間とともに認識度が上がる提示アルゴリズムを複数実装して比較する。候補: 透明度の上昇（フェードイン）、ストローク画素の累積（既存の加算型マスクの動画化）、全候補の重畳から絞り込む減算型（既存 F1 の動画化）、拡大縮小、移動。PowerPoint のアニメーションで実現できる種類を意識する。
- 視覚の2文字課題は「1文字目を 0.2 秒提示して消去し、2文字目を frac% 時点まで提示して消す（または止める）」という、聴覚の時間ゲートの視覚版として設計する。
- フォントの種類（6フォント）も引き続き独立変数の候補として留意する。

## 変わらないもの

- 単音課題（C1 なし）の資産と実験ページは、先頭位置のベースラインとしてそのまま残す。
- 倫理審査は提出済みの内容で進める（追加編集はしない）。
- 分析は2パラメータのロジスティック回帰。混同行列の分析も予定どおり。
- チート対策は標準セット、answer_key の分離、ハッシュ名。
