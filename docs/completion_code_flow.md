# iFont 実験 — 完了コード照合フロー 手順書

> 視覚版 Kikiwake 実験「iFont」における **完了コード(completion code)** の
> 生成→表示→参加者貼付→スプレッドシート記録→照合→承認/否認 までの
> エンドツーエンド手順書です。研究側(リサーチアシスタント)がそのまま使える実務手順としてまとめています。
>
> - 関連: [crowdsourcing_yahoo.md](./crowdsourcing_yahoo.md)
> - 実装根拠: `experiment/experiment.js`(コード生成・送信)、`gas/code.gs`(doPost 記録)
> - 最終更新: 2026-06-06

---

## 1. 全体フロー(エンドツーエンド)

```text
[参加者ブラウザ / experiment.js]
  (1) 実験開始時に 16桁ランダム完了コードを生成
       └ 文字種: A-Z + 2-9(紛らわしい I, L, O, 0, 1 を除外)
  (2) 各トライアル回答ごとに GAS(doPost)へ送信
       送信内容: worker_id, completion_code, stimulus_id,
                 response_char, rt_ms, is_catch, ts ...
  (3) 全200問完走後、完了画面に完了コードを表示
        │
        ▼
[参加者]
  (4) 完了コードをコピーし、Yahoo!クラウドソーシングの回答欄に貼付・提出
        │
        ▼
[Google スプレッドシート(GAS doPost で追記)]
  (5) "trials" シートに 1トライアル=1行 で記録
       列: ts, participant_id, worker_id, completion_code,
           stimulus_id, response_char, correct_char, correct,
           r, font, mode, rt_ms, is_catch
        │
        ▼
[研究側 = あなた]
  (6) 回答欄のコードでスプレッドシートを検索 → 一致確認
  (7) キャッチ正答率・RTフィルタで品質判定
  (8) Yahoo!管理画面で 承認 / 否認
```

**ポイント:** 完了コードは参加者の手元(完了画面)と、研究側のスプレッドシート(各トライアル行の `completion_code` 列)の **両方に同じ値が存在** します。両者の突合が照合の核です。

---

## 2. 各段階の詳細

### (1) 生成 — `experiment/experiment.js`

- 実験ロード時にブラウザ側で **16文字のランダム完了コード** を生成。
- 使用文字: `ABCDEFGHJKMNPQRSTUVWXYZ23456789`(30種)。**読み間違いやすい I・L・O・0・1 を意図的に除外**。
- 同時に作業者識別子 `worker_id` を URL クエリ(`?worker_id=` または `?wid=`)から取得。無い場合は `anon-xxxxxxxx` を自動採番。
  - **【要確認】** Yahoo!クラウドソーシングが URL に作業者IDを自動付与できるか。できない場合は参加者に作業者IDを回答欄へ併記してもらう運用にする。

### (2) 記録 — `gas/code.gs`(doPost)

- 各トライアル回答時にクライアントから POST(`mode: "no-cors"`)。
- GAS が `SPREADSHEET_ID` のシートの `"trials"` に1行追記。
- 記録列(13列):
  `ts, participant_id, worker_id, completion_code, stimulus_id, response_char, correct_char, correct, r, font, mode, rt_ms, is_catch`
- `correct` は送信された `response_char` と `ANSWER_KEY` の正解 `answer` の一致で算出。
- `is_catch` はキャッチ設問フラグ。キャッチ = `r === 100`(完全可視)の刺激。

### (3) 表示 — 完了画面

- 完走時に最後の画面で完了コードを大きく表示(`completion-code` クラス)。
- 参加者IDと所要時間(秒)も併記。
- **途中離脱すると完了画面に到達せず、コードは案内されない。**

### (4) 貼付 — 参加者

- 参加者が完了コードをコピーし、Yahoo!クラウドソーシングの回答欄に貼り付けて提出。

### (5)〜(8) 照合・承認 — 研究側(次節)

---

## 3. 研究側の照合手順(実務ステップ)

> 募集が一区切りついた段階、または定期的(例: 1日1回)にまとめて処理します。

### Step 1. データ準備

1. GAS が記録している Google スプレッドシートの `"trials"` シートを開く。
2. 解析しやすいよう、作業者単位の集計シートを用意(推奨)。**【要確認】** 集計方法は手動ピボット/関数/別スクリプトのいずれでもよい。
   - 例: ピボットテーブルで `completion_code`(または `worker_id`)を行に、以下を集計値にする。
     - トライアル数(完走チェック: 本番200問が揃っているか)
     - キャッチ設問の正答率 = `is_catch=TRUE` かつ `correct=TRUE` の数 ÷ `is_catch=TRUE` の数
     - RT異常率 = `rt_ms<300` または `rt_ms>8000` のトライアル比率

### Step 2. 完了コードの照合

1. Yahoo!管理画面で参加者が提出した完了コードを1件取得。
2. スプレッドシートで **そのコードを検索**(Ctrl/Cmd+F、または `=MATCH()`/`=COUNTIF()`)。
3. 判定:
   - **一致あり** → Step 3(品質判定)へ。
   - **一致なし/空欄** → 否認候補(完了コード不一致)。問い合わせ対応(第6節)を検討。

### Step 3. 品質判定(キャッチ正答率 + RTフィルタ)

該当作業者のトライアル群について:

1. **完走確認** — 本番トライアル数が想定(200問)に達しているか。著しく不足 → 未完走として保留/否認候補。
2. **キャッチ正答率** — `is_catch=TRUE` のトライアルの正答率を算出。
   - 閾値 **【要確認】例: 80%以上で合格**。下回る場合は否認候補。
3. **RTフィルタ** — `rt_ms < 300ms`(早押し)/ `rt_ms > 8000ms`(放置)のトライアル。
   - これらは解析から除外する対象。**異常トライアル比率が高い場合 【要確認】(例: 全体の◯%超)** は否認候補。
4. **ホワイトリスト/チェック設問** — 既存の品質管理(チェック設問・ホワイトリスト併用)とも整合させる。

### Step 4. 承認/否認(Yahoo!管理画面)

- すべての判定をクリア → **承認**(PayPayポイント付与)。
- いずれかに該当 → **否認 または 保留して確認連絡**。否認時は理由を簡潔に記録。
- 処理結果(承認/否認・理由・日時)を作業者単位の管理シートに記録しておくと監査・再発防止に役立つ。

> **運用上の注意:** 軽微・境界的なケースは否認ではなく承認寄りに倒す/個別確認する方針を研究代表者と事前に合意しておく。**【要確認】**

## 4. 判定基準サマリ表

| チェック項目 | 合格条件(案) | 不合格時の扱い |
|---|---|---|
| 完了コード一致 | 回答欄コード = スプレッドシート `completion_code` | 否認候補(不一致) |
| 完走 | 本番トライアル ≈ 200問 | 保留/否認候補(未完走) |
| キャッチ正答率 | **【要確認】例: ≥ 80%** | 否認候補 |
| RT異常率 | 異常トライアル比率 **【要確認】例: < ◯%** | 否認候補(該当トライアルは解析除外) |
| 重複なし | 同一 worker_id/コードの記録が1人分 | 否認候補(重複参加) |

---

## 5. なりすまし・二重応募・コード使い回しへの対策メモ

- **コードとデータの突合を必須化** — 提出コードがスプレッドシートに存在し、かつ十分なトライアル記録を伴うことを確認。コードだけ知っていてデータが無い=なりすまし/未実施を弾ける。
- **コードは1人1回のみ有効** — 同じ完了コードが複数の応募で提出された場合、2件目以降は否認候補。スプレッドシート上で `completion_code` の重複(`COUNTIF`)を監視。
- **worker_id とコードの対応を確認** — 1つの `worker_id` に複数の完了コード、または1つのコードに複数の `worker_id` が紐づく場合は不自然。要確認。
- **再応募不可設定** — プラットフォーム側の「同一作業者の再応募制限」を有効化(crowdsourcing_yahoo.md 設定パラメータ参照)。**【要確認】**
- **コード生成はサーバ管理ではなくクライアント生成** — 現実装はブラウザ側生成のため、理論上は参加者が偽コードを作れる。**ただし偽コードはスプレッドシートのトライアル記録と一致しない**ため、突合で排除可能。これが照合を必須にする最大の理由。
- **タイムスタンプ・所要時間の妥当性** — `ts` の分布や所要時間が極端に短い記録は要注意。
- **個人情報を集めないため、本人性の最終確認はプラットフォームのアカウント制約に依存** する点を理解しておく。**【要確認】**

## 6. 不一致時の対応(問い合わせ対応テンプレ)

> Yahoo!クラウドソーシングのメッセージ機能などで参加者に確認する際のテンプレです。状況に応じて編集してください。

### 6-1. 完了コードが一致しない/見つからない場合

```text
このたびは実験「iFont」へのご協力ありがとうございます。

ご提出いただいた完了コードについて、こちらの記録と照合したところ
一致するデータを確認できませんでした。お手数ですが、以下をご確認のうえご返信ください。

1. 実験を最後の「完了画面」まで進められましたか?
   (完了画面に16桁の完了コードが表示されます)
2. 完了画面に表示されたコードを、そのままコピーして貼り付けていただけますか?
   (前後の空白や改行、似た文字の打ち間違いにご注意ください)
3. 差し支えなければ、参加された日時・おおよその所要時間をお知らせください。

ご返信いただいた内容をもとに再度確認いたします。
ご不便をおかけしますが、よろしくお願いいたします。
```

### 6-2. 未完走と思われる場合

```text
ご協力ありがとうございます。
こちらの記録では、本番の回答が最後まで到達していないように見受けられました。

途中で画面を閉じる・通信が途切れる等があった場合、完了コードが発行されません。
もし最後まで進められた場合は、完了画面に表示されたコードをそのまま貼り付けて
ご返信ください。状況を確認のうえ対応させていただきます。
```

### 6-3. 否認をお伝えする場合(理由付き・丁寧版)

```text
このたびは実験「iFont」へのご応募ありがとうございました。
恐れ入りますが、今回のご提出については下記の理由により承認を見送らせていただきました。

理由: 〔完了コードの不一致 / 回答データが確認できない / 品質基準を満たさない 等〕

実験の趣旨や基準に関するご質問がございましたら、このメッセージにご返信ください。
ご協力いただいたこと、改めて感謝申し上げます。
```

> **【要確認】** 否認の最終判断・連絡文面は、プラットフォーム規約と研究代表者の方針に従ってください。否認はトラブルになりやすいため、境界的なケースは承認寄り/個別確認を基本とします。

## 7. 付録: 照合に使うスプレッドシート列(`gas/code.gs` 準拠)

| 列 | 意味 | 照合での用途 |
|---|---|---|
| `ts` | タイムスタンプ | 参加日時・妥当性確認 |
| `participant_id` | 参加者ID | worker_id 補完(匿名時は anon-) |
| `worker_id` | 作業者ID | 重複参加チェック・突合 |
| `completion_code` | 完了コード(16桁) | **回答欄コードとの照合キー** |
| `stimulus_id` | 刺激ID | — |
| `response_char` | 参加者の回答文字 | — |
| `correct_char` | 正解文字 | 正答判定の根拠 |
| `correct` | 正誤(TRUE/FALSE) | キャッチ正答率算出 |
| `r` | 可視度(100=完全可視=キャッチ) | キャッチ識別 |
| `font` / `mode` | フォント・提示モード | 解析用 |
| `rt_ms` | 反応時間(ms) | RTフィルタ(<300 / >8000) |
| `is_catch` | キャッチ設問フラグ | キャッチ正答率算出 |
